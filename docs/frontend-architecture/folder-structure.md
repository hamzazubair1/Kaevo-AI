# Kaevo AI — Frontend Folder Architecture

> Production-grade directory structure designed for Next.js (App Router), scalability, and feature encapsulation.

---

## 1. High-Level Structure

Following a domain-driven, feature-sliced architecture. This prevents the "monolith" problem where `components/` and `hooks/` become massive, unmanageable dumping grounds as the SaaS application scales.

```text
apps/website/ (or shared package)
├── src/
│   ├── app/                # Next.js App Router (Pages & Layouts)
│   ├── assets/             # Static files (images, SVGs, Lottie JSONs)
│   ├── components/         # Global shared UI components (Atoms, Molecules)
│   ├── config/             # Environment vars, feature flags, constants
│   ├── features/           # Domain-specific modules (The core of the app)
│   ├── hooks/              # Global shared React hooks
│   ├── layouts/            # Global layout wrappers
│   ├── lib/                # 3rd-party library configurations (Axios, Supabase)
│   ├── services/           # Global API clients and endpoints
│   ├── store/              # Global state management (Zustand/Redux)
│   ├── styles/             # Global CSS, Tailwind config, fonts
│   ├── types/              # Global TypeScript interfaces
│   └── utils/              # Pure helper functions
```

---

## 2. Directory Responsibilities

### `src/app/` (Routing Layer)
- **Responsibility:** Maps URLs to React components via the Next.js App Router.
- **Rules:** Files here should be thin wrappers. They import a Feature component or Layout and render it. No business logic belongs in `page.tsx`.
- **Structure:**
  - `(marketing)/page.tsx`
  - `(dashboard)/admin/pipeline/page.tsx`
  - `api/` (Next.js serverless route handlers)

### `src/components/` (Shared UI Layer)
- **Responsibility:** Houses all presentation-only components (Atoms, Molecules, general Organisms).
- **Rules:** Cannot import from `features/`, `store/`, or `services/`. Must be entirely generic.
- **Structure:**
  - `ui/` (Buttons, Inputs, Cards - often generated by shadcn/ui)
  - `form/` (Form wrappers, specific styled inputs)
  - `feedback/` (Spinners, Toasts, Skeletons)

### `src/features/` (Domain Layer - CRITICAL)
- **Responsibility:** Encapsulates everything related to a specific business domain.
- **Rules:** A feature can import from `components/` and global `utils/`, but features should rarely import from *other* features unless explicitly designed as a shared feature.
- **Structure Example (`features/ai-chat/`):**
  - `components/` (Chat-specific UI: ChatBubble, SuggestionChips)
  - `hooks/` (e.g., `useChatStream.ts`)
  - `services/` (API calls specific to the chat domain)
  - `store/` (Local state slice for the chat widget)
  - `types/` (Chat-specific interfaces)
  - `index.ts` (Public API exporting only what other parts of the app need)

### `src/lib/` (Infrastructure Layer)
- **Responsibility:** Configuration and initialization of external libraries.
- **Examples:** `supabase.ts` (client init), `axios.ts` (interceptors), `react-query.ts` (provider config).

### `src/services/` (API Layer)
- **Responsibility:** Functions that make HTTP requests. 
- **Rules:** Should be pure async functions that return promises. Data fetching hooks (React Query) belong in `features/` or `hooks/`, which *call* these services.

### `src/store/` (Global State Layer)
- **Responsibility:** Client-side state that MUST be shared across disparate features (e.g., Current User Session, Global Theme).
- **Rules:** Keep as thin as possible. Server state belongs in React Query cache, not here.

## 3. Scalability Considerations

### Multi-Tenant System
- The structure supports multi-tenancy by keeping domain logic in `features/`. When transitioning to a full SaaS platform (Phase 5), we can extract features like `features/crm` or `features/ai-chat` into separate packages within a Turborepo to share across multiple distinct applications.

### AI Agents Implementation
- AI interactions are treated as standard features (`features/ai-receptionist`, `features/ai-qualification`).
- This allows us to cleanly swap out or version AI agents without touching the global UI components or routing layer.

### Role-Based Access
- Handled at the `app/` level using Next.js Middleware and Layouts.
- e.g., `app/(dashboard)/admin/layout.tsx` validates the admin role before rendering any children.
