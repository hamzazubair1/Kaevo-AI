# Kaevo AI — Design System to Component Mapping

> Translates the UI/UX design system (Day 3) into a structured hierarchy of reusable frontend components.

---

## 1. Atomic Component Strategy

We follow the Atomic Design methodology to build UI elements that are highly reusable, strongly typed, and isolated from business logic.

### Atoms
The smallest, indivisible UI elements. Built strictly from design tokens.
* **Typography:** `<Heading />`, `<Text />`, `<Label />`
* **Inputs:** `<Input />`, `<TextArea />`, `<Checkbox />`, `<Toggle />`, `<Radio />`
* **Buttons:** `<Button />` (handles variants: Primary, Secondary, Ghost, Destructive, Loading)
* **Visuals:** `<Icon />`, `<Avatar />`, `<Badge />`, `<Tooltip />`, `<Spinner />`
* **Layout Primitives:** `<Flex />`, `<Grid />`, `<Container />`, `<Divider />`

### Molecules
Simple combinations of atoms working together.
* **Form Groups:** `<FormField />` (combines `<Label />`, `<Input />`, `<ErrorText />`)
* **List Items:** `<ListItem />` (combines `<Icon />`, `<Text />`)
* **Search:** `<SearchBar />` (combines `<Input />`, search `<Icon />`)
* **Chips:** `<SuggestionChip />` (combines ghost `<Button />` styling with specialized click behavior)
* **Toast:** `<Toast />` (combines `<Icon />`, `<Text />`, close `<Button />`)

### Organisms (Composite Components)
Complex, distinct sections of an interface that function independently but are still presentation-only.
* **Cards:** 
  * `<MetricCard />` (Overview KPIs)
  * `<LeadCard />` (CRM Pipeline)
  * `<ServiceCard />` (Marketing site offerings)
* **Forms:** `<QualificationForm />`, `<AuthForm />`
* **Modals/Drawers:** `<ConfirmModal />`, `<SideDrawer />`
* **Tables:** `<DataTable />` (combines headers, rows, pagination, sorting)
* **Navigation:** `<TopNavbar />`, `<Sidebar />`, `<MobileMenu />`
* **Chat:** `<ChatMessage />` (combines `<Avatar />`, `<Text />`, timestamp, background styling)

---

## 2. Feature Components

Feature components bind Organisms to business logic, state management (Redux/Zustand), and data fetching (React Query). They are domain-specific.

### AI Engine Features
* **`<AIChatWidget />`:** The entire floating receptionist widget (handles WebSocket connection, message streaming, intent routing).
* **`<AIFullScreenChat />`:** The dedicated `/chat` view for deep qualification.
* **`<InteractiveProposalCard />`:** Renders the customized proposal generated by Agent A3/B4.

### Dashboard Features
* **`<LeadPipelineBoard />`:** The drag-and-drop Kanban interface for CRM.
* **`<TelemetryDashboard />`:** The client-facing ROI visualization grid.
* **`<AgentApprovalQueue />`:** The table interface for admins to review and approve Agent B5's drafted emails.
* **`<MeetingScheduler />`:** The embedded calendar integration (Cal.com wrapper) that communicates booking success back to the parent state.

---

## 3. Layout Components

Layouts define the macroscopic structure of a page and persist across route changes.

* **`<MarketingLayout />`:** Header (sticky), Main Content `children`, Footer. Standard max-width constraints.
* **`<DashboardLayout />`:** Fixed Sidebar (or bottom nav on mobile), Top bar (search/user profile), scrollable Main Content area.
* **`<AuthLayout />`:** Split screen (branding graphic on left, form centered on right).
* **`<FocusLayout />`:** Minimal header (logo only), no footer. Used for the dedicated AI Chat (`/chat`) and Meeting Booking to remove distractions.

---

## 4. Component Implementation Guidelines

To ensure scalability and consistency across the monorepo:

1. **Strict Separation:** Atoms, Molecules, and Organisms MUST be "dumb" (presentation only). They receive data via props and emit events via callbacks.
2. **Feature Encapsulation:** Only Feature components should hook into global state or API services.
3. **Styling:** Use a utility-first CSS framework (Tailwind) bounded by a strict configuration file derived directly from `design-tokens.md`. Avoid arbitrary values (e.g., use `p-6` not `p-[24px]`).
4. **Prop Types:** All components must be strictly typed with TypeScript interfaces.
5. **Variants:** Use a variant management library (like `cva` - class-variance-authority) to cleanly handle complex component states (e.g., Button variants, sizes, and disabled states).
